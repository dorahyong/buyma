# 통합 자동화 (비개발자용 정리)

이번 작업은 크게 **두 축의 고도화**.
- **A. 중복 고도화** — *무엇을* BUYMA에 올릴지 (같은 품목을 어떻게 묶고 어느 소싱처로 올릴지)
- **B. 자동화 고도화** — *어떻게* 돌릴지 (그 처리를 끊김 없이, 효율적으로, 이력 남기며)

(기술 상세: `unified_automation_DEV_20260623.md`)

---

## 1. 목표

> 여러 소싱처에서 수집된 중복 품목을 BUYMA에 1개만 올리되,
> 제일 싼 소싱처의 데이터를 사용하고, 출품 가능한 옵션(색,사이즈 등)은 모두 모아서 올린다.

---

# A. 중복 고도화 — 무엇을 올릴지

## A-1. 예전 방식: "죽이기"
같은 품목이 여러 소싱처에서 수집되면, **우선순위 소싱처 하나만 남기고 나머지는 비활성화(죽임)**.
- 한계 1: 더 싼 소싱처를 죽여버려 **비싸게 마진계산**
- 한계 2: 소싱처마다 가진 색·사이즈가 다른데 **하나만 남겨서 옵션 못 모음**.

## A-2. 지금 방식: "병합(reconcile)"
같은 품번을 여러 소싱처에 걸쳐 **하나의 BUYMA 출품으로 병합**.
- **제일 싼 소싱처**를 골라 그 데이터로 출품, 마진계싼 (winner).
- 출품 가능한 **옵션(색·사이즈)을 전부 모아서**(합집합) 올림.
- 마진이 안 나거나 전체 품절이면 출품 안 함 / 내림.

신규 등록이든, 기존 상품 재고관리든 **둘 다 같은 병합 로직 사용**.

## A-3. 효과
- 같은 품목을 **더 싸게 마진계산**, **더 많은 옵션**으로 → 경쟁력↑, 판매↑.
- "죽여서 버리던" 소싱처 정보를 **살려서 활용**.

## A-4. (참고) 현재 "같은 품목"으로 묶는 기준 — 그룹핑

두 수집 데이터가 **같은 그룹(=같은 품목)** 으로 묶이는 기준 (현재):

- **같은 브랜드** + **품번(model_no)을 정규화한 값이 같음**.
- 품번 정규화 규칙:
  1. 끝의 괄호 코드 제거 (예: `ABC123(NOO)` → `ABC123`)
  2. `A / B` 처럼 슬래시로 두 개 병기돼 있으면 **뒤의 것만** 사용
  3. 앞의 시즌 코드 제거 (예: `25FW-ABC` → `ABC`)
  4. 공백·하이픈·슬래시·따옴표 제거 + 대문자 (예: `abc-123` → `ABC123`)
  → 이렇게 다듬은 값이 같으면 같은 품목으로 봄.
- 추가 규칙(주의): 같은 브랜드 안에서 한 품번이 다른 품번에 **포함**되면 같이 묶음 (예: 짧은 코드 `V0020` 이 여러 긴 품번 안에 들어가면 다 묶임).

> 추후 그룹핑 고도화 필요

# B. 자동화 고도화 — 어떻게 돌릴지

buyma 에 5만개를 채우는 것을 목표로 자잘한 이슈는 넘어가고, 특히 무지성 중복 검증을 위한 로직 떄문에 정합성이 많이 떨어지는 상태여서 고도화 진행.

자동화 쪽도 예전엔 소싱처 묶음별로 **명령 3개**(okmall / 멀티소스 / 네이버)를 따로 돌렸고, **이력 없음 · 중단되면 처음부터 · 느린 소싱처가 전체를 막는 병목 · 신규/재고를 따로 돌림** 같은 이슈들을 그냥 넘긴 상태였음. 이번에 하나로 합침.

## B-1. 합치기
- 명령 3개 → **`run_daily_unified.py` 한 개**로 통합. **31개 소싱처**(okmall 1 + 멀티소스 9 + 네이버 21) 전부 한 번에.

## B-2. 이력 남기기
- **어느 소싱처가 어느 단계까지 진행됐는지 / 어디서 실패했는지** 기록(DB).
- 예전엔 "지금 어디까지 됐나"를 알 수 없었는데, 이제 **현황 파악** 가능.

## B-3. 이어돌기 (이미 한 일은 다시 안 함)
- 중간에 멈추거나 껐다 켜도 **처음부터 다시 하지 않고**, 끝낸 단계는 건너뛰고 **멈춘 데서 이어서** 진행.
- 예전엔 중단되면 처음부터라 하루에 못 끝내고 매번 실패하던 문제가 있었는데, 그게 해소.

## B-4. 순서 효율 (병목 제거)
- 네이버의 경우 중복 검증을 위해 수집 병목을 의도적으로 넣은 채로 진행해서 속도가 많이 느렸는데, 이 부분을 모두 개선

## B-5. 신규·재고 동시 + 네이버만 직렬
- **신규 등록 갈래**(아직 BUYMA에 없는 품목)와 **재고 관리 갈래**(이미 올라간 품목)를 **동시에** 진행.
- 단, **네이버만 예외** — 캡챠(로봇 확인) 때문에 동시에 여러 개가 접속하면 막혀서, **네이버는 사이트 접속(수집·재고)을 한 번에 하나씩**만 진행. (나머지 소싱처는 다 동시)

---

## 2. 가격도 통일

- **경쟁자가 있으면**: 경쟁자보다 **1~9엔 싸게**.
- **경쟁자가 없으면**: 매입가 기준 **마진 30%**.

---

## 3. 어떻게 돌리나

```
python run_daily_unified.py
```
- 한 번 실행 = **한 바퀴** (31개 소싱처의 모든 단계를 다 하면 종료).
- **끝나면 또 실행** → 계속 반복하면 상시 자동화. (지금은 수동, 나중엔 자동 반복)
- 중간에 꺼져도 다시 켜면 **이어서** (B-3).
- ※ 네이버는 로그인 쿠키 필요(사용자 관리).

> "기존 5만 개를 한 번에 정리하는 작업"은 **불필요**. 시스템이 계속 돌면 재고관리 갈래가 **매 바퀴 전체를 다시 갱신**하므로 자동으로 정리.

---

## 4. 검증된 것 (실제 BUYMA 반영)

- 등록~업데이트 전체 사이클 테스트 완료 (mall 1개)
- 이어돌기·병렬·네이버 직렬 등 안전장치 테스트 통과.
- 전체 점검 결과 치명적 버그 없음(자잘한 것들은 발견 즉시 수정).

---

## 5. 남은 일 (1가지)

### 데이터 구조 정리 — 나중에, 천천히, 단 고위험
- 지금도 정상 작동. "고장 수리"가 아니라 "정리정돈".
- "이 품목이 BUYMA에 있나" 정보가 두 군데 나뉜 걸 한 곳으로 모으는 일. 광범위·고위험이라 **충분히 검증된 뒤 맨 마지막**.

---
